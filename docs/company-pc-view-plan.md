# 会社PC向けビュー改善計画

## 目的

この文書は、会社支給のノートPCで使うことを前提にした専用レイアウト方針をまとめたものです。

現在のメインUIは、広めのモニターでは成立していても、会社ノートPCの100%表示では窮屈に見えたり、情報が詰まりすぎたりします。

そのため、ここでは「単なる縮小版」ではなく、**会社PC向けの専用ビュー**として考えます。

## なぜ専用ビューが必要か

このプロジェクトの現行UIは、広いモニターでは見やすくても、会社ノートPCの幅では余白や横並びがきつくなります。

実際に確認したノートPCの値は次の通りです。

- `window.innerWidth`: `1258px`
- `document.documentElement.clientWidth`: `1243px`
- `window.devicePixelRatio`: `1.50`

この幅は、一般的な `xl` ブレイクポイントの手前にあたります。  
つまり、**「広いけれど、まだ大画面専用の並べ方だと厳しい」帯**です。

## 目標

会社PC向けビューでは、次を満たしたいです。

- 地図が読みやすい
- 右側の会話欄が使いやすい
- 画面横幅に無理をさせない
- 会社PCの幅帯で安定して見える
- 後で消したり差し替えたりしやすい

## 対象外

この計画では、次まではやらない想定です。

- 全端末向けの大規模な再設計
- モバイル最優先のデザイン変更
- 別リポジトリ化
- 機能追加を先に増やすこと

## レイアウト方針

会社PC向けでは、**横の広がりよりも見やすさを優先**します。

考え方は次の通りです。

- 大画面でも、右のチャット欄がきちんと活きる分割に寄せる
- 会社PCでも、その分割のほうが見やすく使いやすい
- 情報を「並べる」より「役割ごとに見せる」方向に寄せる

会社PC向けビューは、縮めたデスクトップ画面ではなく、**会社PC用に組み直した画面**として扱います。

このときの基本イメージは、次のようなものです。

- 左: 地図
- 右: チャット欄
- 上: 最小限の説明

つまり、単なるレスポンシブ調整ではなく、**右側のチャット欄が主役として見える分割方法**に変えます。

## おすすめの構成

### 基本構成

会社PC向けでは、1列寄り、または弱い2列構成にします。

優先順位は次の通りです。

1. タイトルと選択中エリア
2. 地図
3. 会話欄
4. 補助カードや説明

ただし、画面全体の方針としては、右のチャット欄が十分に見える2分割を基本に考えます。  
大きい画面でも、会社PCでも、チャット欄が押しつぶされない分割を優先します。

### 地図エリア

地図は主役のままにしますが、固定幅の重いポップアップや、広い前提の装飾は減らします。

調整候補:

- ポップアップの固定幅を小さくする
- ポップアップのカード自体を小さくして、地図の露出面積を増やす
- アイコンなどの視覚的な手がかりは残す
- 地図見出しの高さを抑える
- 地図が画面でいちばん目立つようにする

### 会話エリア

会話欄は、見えること自体が大事ですが、横幅を食いすぎないようにします。

調整候補:

- 見出しは大きくしすぎない
- 本文の基準サイズも落とす
- 全体のジャンプ率を小さくして、画面の起伏を穏やかにする
- 会話欄は縦スクロール前提にする
- 高さ固定で押しつぶされる形を避ける
- 見出し周りの装飾を減らす

### ヘッダー

上部のヘッダーは、かなり薄くしてよいです。最悪、なくても構いません。

調整候補:

- タイトルと説明だけに絞る
- 3つのカードは削除する
- 余白を減らして、地図と会話欄に情報面積を回す
- 右側のチャット欄が見やすくなるよう、上部で場所を使いすぎない

### 文字サイズ

このビューポートでは、大きな見出しは元気があっても情報の邪魔になります。

調整候補:

- 見出しを小さくする
- 本文も小さくする
- ポップアップカードも同じ考え方で小さくする
- 文字の強弱を抑え、全体のジャンプ率を小さくする

### 分割方法

画面の分割方法そのものも見直します。

考え方:

- 左の地図を広く取りすぎない
- 右のチャット欄を、会話しやすい幅で確保する
- 上部の要素は最小限にする
- 会社PCでも、大画面でも、この役割分担を崩さない

要するに、`AS IS` のように地図が強すぎる構成ではなく、`TO BE` のように**右のチャット欄が活きる分割**に寄せます。

## ブレイクポイント方針

会社PCの幅帯を、最初から重要な前提として扱います。

目安となる幅帯:

- `base`: `768px` 未満
- `md`: `768px` 以上
- `lg`: `1024px` 以上
- 会社PC帯: おおよそ `1200px` 〜 `1280px`
- `xl`: `1280px` 以上

今回のノートPCは `1258px` 前後なので、**会社PC帯のど真ん中**に近いです。  
そのため、`xl` 前提の見せ方のままだと、どうしても厳しくなります。

## まず直すべきこと

### 1. 固定幅の前提を減らす

次のような要素を見直します。

- 固定幅の大きいポップアップ
- 右レールの幅前提
- ヘッダー内の大きめカード
- 大きな見出しと大きすぎる本文
- `xl` 前提の横並び
- 地図が広すぎて右のチャット欄を圧迫する分割

### 2. メイングリッドを見直す

会社PCでまだ窮屈なら、2列にこだわりすぎず、縦の流れを強めます。

### 3. 右側パネルを軽くする

右レールが細くなりすぎる場合は、次だけを残します。

- 選択中のエリア
- 質問入力
- 会話ログ

補助的な見た目は外に逃がします。

### 4. 地図の操作性を守る

地図は最後まで読めること、押せることが重要です。

他の要素が地図にかぶりすぎないようにします。

## 実装方法の候補

### 案A. 1つのアプリの中で分岐する

画面幅を見て、会社PC用のレイアウトに切り替える方法です。

良い点:

- 管理が楽
- ひとり作業に向いている
- 追加の導線が少ない

注意点:

- 分岐が増えると、コードが少し複雑になる

### 案B. 会社PC用の別ルートを作る

会社PC専用の画面を、別ページとして持つ方法です。

良い点:

- 本体UIと明確に分けられる
- 既存の大画面UIを壊しにくい

注意点:

- 画面が増える
- 運用導線が少し増える

### おすすめ

最初は、**別ルートまたは別ページ**として作るのが分かりやすいです。  
会社PC専用であることが明確になり、今のメインUIに悪影響を与えにくいからです。

もし後で差が小さくなったら、1つのアプリに統合してもかまいません。

## 検証方法

会社PCを、最優先の確認端末として使います。

確認したい状態:

1. ブラウザ 100%
2. ブラウザ 90%
3. エリア選択ポップアップ表示時
4. 会話が数件たまった状態
5. 地図がまだ十分に読める状態

確認の合否基準:

- ヘッダーが切れない
- 地図と右レールが大きく重ならない
- 会話欄が読める
- ブラウザの縮小に頼らなくても使える

## 次の進め方

1. いまの `debug-width` ページを測定用として残す
2. 会社PC向けレイアウトの要件を固める
3. 別ページにするか、条件分岐にするか決める
4. まずは最初の会社PC向けレイアウトを作る
5. 会社PCでブラウザ100%のまま再確認する

## 補足

- この計画は、観光PoC本体とは切り分けて考えます。
- `debug-width` ページは、幅測定用としてそのまま使えます。
- 会社PC向けレイアウトは、見た目の好みではなく、業務上の使いやすさとして扱います。
